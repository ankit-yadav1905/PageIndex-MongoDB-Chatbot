\section{Introduction}
The objective of this project is to develop a sophisticated, vectorless Retrieval-Augmented Generation (RAG) chatbot. Traditional RAG architectures rely on vector databases and cosine similarity to retrieve chunked text. While effective for general semantic search, this approach often fails when querying complex, structured documents due to lost context. 

This project implements a novel decoupled architecture utilizing the \textbf{PageIndex} framework, which leverages Large Language Model (LLM) reasoning over hierarchical document trees, supported by a \textbf{MongoDB} system of record, and a \textbf{Streamlit} user interface for continuous conversational context.

\section{Key Conceptual Learnings}
During the development of this project, several core architectural concepts were mastered:

\subsection{Traditional Vector RAG vs. Vectorless RAG (PageIndex)}
\begin{itemize}
    \item \textbf{Traditional RAG:} Typically involves chunking text into small overlapping pieces, converting them into high-dimensional vectors, and storing them locally. While fast, it often loses the overarching context or structural meaning of long documents, returning mathematically similar but contextually irrelevant chunks.
    \item \textbf{Vectorless RAG:} Uses LLMs to read the document and build a hierarchical tree of nodes remotely. When querying, it logically traverses this tree to find the exact section containing the answer, preserving deep structural context without relying on vector similarity math. This offloads the heavy lifting to external servers.
\end{itemize}

\subsection{Why use MongoDB? (Decoupled Architecture)}
A major conceptual breakthrough was understanding why a local database is necessary when the heavy text processing is handled by PageIndex. 
\begin{itemize}
    \item \textbf{Multi-tenancy \& Security:} PageIndex has no concept of "users". MongoDB acts as the system of record to map specific documents to specific users, ensuring privacy in a production environment.
    \item \textbf{Rich Custom Metadata:} PageIndex only stores raw text. MongoDB allows the application to store custom descriptions, timestamps, author names, and custom tags alongside the simple \texttt{doc\_id} returned by PageIndex.
    \item \textbf{Performance Optimization:} Querying a local database to populate the UI (e.g., listing all uploaded books) is instantaneous and avoids hitting third-party API rate limits.
\end{itemize}

\section{System Architecture}

\subsection{MongoDB: The System of Record}
MongoDB serves as the foundational metadata store for the application, deployed via \textbf{Docker}.
\begin{itemize}
    \item \textbf{Containerization:} A \texttt{docker-compose.yml} file orchestrates the \texttt{mongo:latest} image.
    \item \textbf{Role in RAG:} MongoDB stores a collection named \texttt{document\_nodes}. When a document is ingested, MongoDB records the \texttt{filename}, \texttt{upload\_date}, \texttt{description}, and crucially, the \texttt{pageindex\_doc\_id}.
\end{itemize}

\subsection{PageIndex: Vectorless Reasoning Engine}
PageIndex acts as the core retrieval engine. The workflow operates as follows:
\begin{enumerate}
    \item \textbf{Document Ingestion:} Documents are submitted to the PageIndex API via the \texttt{submit\_document} endpoint.
    \item \textbf{Hierarchical Tree Construction:} The underlying LLM extracts the structural hierarchy of the document remotely.
    \item \textbf{Retrieval via Reasoning:} When a user query is received, the LLM traverses the tree logically to formulate highly accurate, cited answers.
\end{enumerate}

\section{Data Flow}

\subsection{Universal Ingestion Pipeline}
The ingestion process, handled by \texttt{ingest\_document.py}, has been abstracted to be 100\% universal. It uses Python's \texttt{argparse} to accept any PDF from the command line without hardcoded paths:
\begin{lstlisting}[language=Python]
# 1. Terminal Execution
# python src\ingest_document.py "path/to/book.pdf" --description "Optional"

# 2. Submit document to PageIndex
result = pi_client.submit_document(args.file_path)
doc_id = result.get("doc_id")

# 3. Store metadata in MongoDB
document_record = {
    "filename": os.path.basename(args.file_path),
    "description": args.description,
    "pageindex_doc_id": doc_id,
    "status": "indexed"
}
db_manager.insert_node(document_record)
\end{lstlisting}

\subsection{Retrieval and Chat Pipeline}
The retrieval process is triggered by the user via the Streamlit interface:
\begin{lstlisting}[language=Python]
# 1. Retrieve doc_id from MongoDB based on selected filename
record = db_manager.collection.find_one({"filename": doc_name})
doc_id = record.get("pageindex_doc_id")

# 2. Query PageIndex with full conversation history
response = pi_client.chat_completions(
    messages=messages_history,
    doc_id=doc_id
)
\end{lstlisting}

\section{User Interface (Streamlit)}
The application layer is built using Streamlit (\texttt{app.py}).
\begin{itemize}
    \item \textbf{Dynamic Selection:} The sidebar queries MongoDB dynamically to populate a dropdown menu of all universally uploaded documents.
    \item \textbf{Continuous History:} Utilizing \texttt{st.session\_state}, the application stores the array of \texttt{messages}. This allows the PageIndex reasoning engine to maintain conversational context across multiple sequential questions.
\end{itemize}

\section{Conclusion}
By combining the structural metadata flexibility of MongoDB, the heavy-duty reasoning capabilities of PageIndex, and a dynamic UI in Streamlit, this project successfully implements a highly scalable, decoupled architecture. This design mirrors production-grade enterprise SaaS platforms, proving superior to traditional vector databases for complex, unstructured data comprehension.
